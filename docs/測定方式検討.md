# 測定方式検討（tsuzumi2 vs AOAI）

## 0. 目的
* tsuzumi2（tsuzumi28B / 28B量子化）と Azure OpenAI（例：GPT-4o）を、**同一I/F・同一プロンプト・同一評価軸**で比較する
* 対象タスク：前処理／深堀分析（レポート化）／質問（QA）の3パターン

> **スキーマ上の `task_type` enum 値との対応**
> | 本文書での呼称 | `task_type` enum |
> |---|---|
> | 前処理 | `preprocessing` |
> | 深堀分析（レポート化） | `report_generation` |
> | 質問（QA） | `report_qa` |

---

## 1. 共通前提（ブレ要因の排除）
### 1.1 API仕様
* tsuzumi2 APIは **OpenAI API: Create chat completion** 互換として扱う  
  参照： https://platform.openai.com/docs/api-reference/chat
* 原則：**Chat Completions形式**（`model`, `messages`, 主要生成パラメータ）で統一

### 1.2 リクエスト固定方針（比較のためのルール）
* 原則固定（全モデル共通に揃える）
  * `messages`（system含む）、入出力フォーマット、最大出力長（`max_tokens`）
  * `temperature` / `top_p`（基本は temperature=0 など、要合意）
  * ストップ条件（`stop` を使うなら同一に）
* 可変（実験条件として掃引する場合のみ）
  * 同時接続数（concurrency）
  * 1ユーザあたりのリクエスト頻度（RPS）
  * 入力長（短文/中文/長文）※必要ならバケット化
* 再現性
  * `seed` 等が両方で使える場合のみ固定。使えない場合は試行回数を増やし統計で吸収

### 1.3 計測ログ（最低限）
* request_id, timestamp, model, prompt_version, パラメータ一式
* input_tokens / output_tokens（取得できる範囲で）
* latency（end-to-end）と、可能なら内訳（queue/compute 等）
* HTTP status / error 種別（rate limit, timeout, 5xx 等）

## 2. 精度比較（Quality）
### 2.1 評価データ
* パターン別にテストケースを作成
  * 前処理：入力（生テキスト/表/ログ等）→ 期待する正規化/要約/抽出項目
  * 深堀分析：入力（一次情報）→ レポート構造（見出し、根拠、結論、留意点）
  * 質問：質問＋コンテキスト → 正答/根拠/禁止事項（推測の断定禁止等）
* 各ケースにメタ情報を付与
  * 難易度、入力長、期待形式、禁止事項、評価観点

### 2.2 定量（LLM-as-a-Judge）
* Judgeは固定（例：AOAI側の特定モデル）し、評価プロンプトも固定
* 1ケースあたり：
  * (A) tsuzumi出力、(B) AOAI出力 を同一条件で生成
  * judgeに **ペア比較**（どちらが要件を満たすか）＋ **理由** を出させる
* スコア例（要調整）
  * Correctness（事実/整合性）
  * Format compliance（指定フォーマット厳守）
  * Safety/Policy（不適切出力・注入耐性観点）
  * Evidence（根拠の明示、参照整合）
* バイアス対策（最低限）
  * 出力A/Bの提示順をランダム化
  * 同一ケースを複数回（n回）実行し分散を見る

### 2.3 定性レビュー
* レビュー項目（会議で合意してチェックリスト化）
  * 内容が合っているか
  * 出力形式が合っているか
  * 不適切な出力がないか
  * セキュリティ的に問題ないか（インジェクション、機微情報混入）
* 重大バグの定義（例：致命誤り、ハルシネで断定、機微漏洩、フォーマット崩れ）を先に決める

## 3. 性能比較（Performance）
### 3.1 指標（必須）
* latency：p50 / p95 / p99（end-to-end）
* throughput：requests/sec、tokens/sec（可能なら output_tokens/sec と total_tokens/sec）
* 成功率：2xx率、タイムアウト率、429率、5xx率
* 安定性：連続稼働時の劣化（例：30〜60分のソーク）

### 3.2 試験条件
* 同一プロンプト・同一パラメータ・同一クライアント実装（OpenAI互換）で統一
* 入力長バケット（例）
  * S: 〜1k tokens、M: 〜4k、L: 〜8k（実データに合わせて調整）
* 出力長
  * `max_tokens` を固定し、短い応答に偏らないよう評価ケース側でも誘導（必要なら）

### 3.3 負荷のかけ方（例）
* warm-up：数分（キャッシュ/初期化影響を除外）
* 本計測：
  * concurrency を段階的に上げる（例：1, 2, 5, 10, 20, ...）
  * 各段階で一定時間（例：5〜10分）計測
* 失敗条件（打ち切り基準）
  * p95 latency が閾値超過、または成功率が一定未満、またはOOM等の致命障害

### 3.4 記録フォーマット（最低限）
* 条件：model / concurrency / input_bucket / max_tokens / temperature / run_id
* 結果：p50/p95/p99、rps、tokens/sec、error率

## 4. 費用比較（Cost）
### 4.1 AOAI（従量課金）の試算
* 月間コスト ≒ Σ（input_tokens × 入力単価 + output_tokens × 出力単価）
* 前提：月間リクエスト数、平均input/output tokens、ピーク/平均RPS、再試行率

### 4.2 tsuzumi（オンプレ / AWS）の試算
* オンプレ
  * ハード（GPU/CPU/メモリ/ストレージ）償却、電力、保守、ラック/設置、運用工数
  * **同時アクセスで性能劣化しない上限ユーザ数**で按分（性能試験結果を使用）
* AWS
  * インスタンス（GPU）、ストレージ、ネットワーク、運用（監視/ログ）等
* 重要：**「同一SLO（例：p95 latency）」を満たす構成で比較**（安いが遅い、を避ける）

### 4.3 メンテ費用の項目（列挙レベル）
* モデル更新、セキュリティパッチ、監視、障害対応、キャパ計画、バックアップ、監査対応
* ※算出はベストエフォート、前提を明記してレンジで提示
